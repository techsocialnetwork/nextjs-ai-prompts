# AI Ethics: Building Responsible AI Systems

AI ethics isn't just about avoiding harm—it's about actively building systems that are fair, transparent, and beneficial to society. Here's what I learned from IBM's comprehensive AI ethics curriculum.

## Why AI Ethics Matters

### Real-World Consequences

**Healthcare Bias**
- Algorithms under-recommending care to Black patients
- Diagnostic tools performing poorly on underrepresented groups
- Life-or-death decisions affected by biased training data

**Criminal Justice**
- Risk assessment tools showing racial disparities
- Predictive policing reinforcing historical biases
- Sentencing recommendations influenced by demographics

**Employment**
- Resume screening tools discriminating by gender
- Facial recognition failing on darker skin tones
- Credit algorithms perpetuating economic inequality

### Business Impact

- **Reputation damage**: Public backlash against biased AI
- **Legal liability**: Regulatory fines and lawsuits
- **Lost opportunities**: Excluding customer segments
- **Trust erosion**: Users abandoning unfair systems

## Core Ethical Principles

### 1. Fairness

**Types of Fairness**

*Demographic Parity*
- Equal positive outcome rates across groups
- Example: Loan approval rates equal across races
- Challenge: May require different thresholds

*Equal Opportunity*
- Equal true positive rates across groups
- Example: Equal job offer rates for qualified candidates
- Focuses on qualified individuals only

*Individual Fairness*
- Similar individuals receive similar treatment
- Requires defining "similarity"
- Challenge: Which features matter?

**Measuring Fairness**

```python
# Disparate Impact Ratio
ratio = (positive_rate_group_a / positive_rate_group_b)
fair = 0.8 <= ratio <= 1.25  # 80% rule

# Equal Opportunity Difference
diff = tpr_group_a - tpr_group_b
fair = abs(diff) < 0.05  # Within 5%
```

### 2. Transparency

**Explainable AI (XAI)**

*Why Explanations Matter*
- Regulatory compliance (GDPR "right to explanation")
- User trust and adoption
- Debugging and improvement
- Accountability

*LIME (Local Interpretable Model-agnostic Explanations)*
- Explains individual predictions
- Creates local surrogate models
- Shows feature importance

```python
from lime.lime_tabular import LimeTabularExplainer

explainer = LimeTabularExplainer(
    training_data,
    feature_names=feature_names,
    class_names=['denied', 'approved'],
    mode='classification'
)

explanation = explainer.explain_instance(
    instance,
    model.predict_proba
)
```

*SHAP (SHapley Additive exPlanations)*
- Game theory-based feature attribution
- Consistent and locally accurate
- Works for any model type

```python
import shap

explainer = shap.TreeExplainer(model)
shap_values = explainer.shap_values(X)
shap.summary_plot(shap_values, X)
```

**Model Cards**

Standardized documentation including:
- Intended use and users
- Training data description
- Performance metrics across groups
- Known limitations
- Ethical considerations

### 3. Privacy

**Differential Privacy**

Mathematical guarantee that individual data doesn't significantly affect output:

```python
# Add calibrated noise
noise = np.random.laplace(0, sensitivity/epsilon)
private_result = true_result + noise
```

- Epsilon (ε): Privacy budget (smaller = more private)
- Sensitivity: Maximum change one record can cause
- Composition: Privacy budget depletes with queries

**Federated Learning**

Train models without centralizing data:

1. Server sends model to clients
2. Clients train on local data
3. Clients send only model updates
4. Server aggregates updates
5. Repeat until convergence

Benefits:
- Data never leaves devices
- Reduced data transfer
- Compliance with data residency laws

**Homomorphic Encryption**

Perform computation on encrypted data:
- Encrypt sensitive data
- Process without decryption
- Results remain encrypted
- Decrypt only final output

## Bias Detection and Mitigation

### Sources of Bias

**Data Collection Bias**
- Historical discrimination encoded in data
- Sampling bias (underrepresentation)
- Measurement bias (faulty sensors)
- Confirmation bias (seeking supporting data)

**Algorithmic Bias**
- Feature selection bias
- Proxy variables (zip code → race)
- Optimization objectives
- Threshold selection

**Deployment Bias**
- Different contexts than training
- Feedback loops
- User interaction patterns
- Concept drift

### Detection Techniques

**Statistical Parity Tests**
```python
def demographic_parity(y_pred, sensitive_attr):
    groups = np.unique(sensitive_attr)
    rates = []
    for group in groups:
        mask = sensitive_attr == group
        rate = y_pred[mask].mean()
        rates.append(rate)
    return max(rates) - min(rates)  # Should be < threshold
```

**Disparate Impact Analysis**
- Compare positive outcome rates
- 80% rule: No group below 80% of max rate
- Legal standard in US employment

**Intersectional Analysis**
- Multiple protected attributes
- Subgroups within groups
- Compounding disadvantages

### Mitigation Strategies

**Pre-processing**
- Reweighting training samples
- Resampling (oversample/undersample)
- Synthetic data generation (SMOTE)
- Feature removal/modification

**In-processing**
- Adversarial debiasing
- Fairness constraints in optimization
- Multi-task learning with fairness objective
- Regularization for fairness

**Post-processing**
- Threshold adjustment per group
- Calibration for equal performance
- Reject option classification
- Human-in-the-loop review

## AI Governance in Practice

### Risk Assessment Framework

**Risk Categories**
1. **Critical**: Life-or-death decisions (healthcare, autonomous vehicles)
2. **High**: Significant impact on rights/opportunities (hiring, lending)
3. **Medium**: Moderate impact (recommendations, customer service)
4. **Low**: Minimal impact (spam filters, content suggestions)

**Assessment Criteria**
- Potential for harm
- Scale of impact
- Reversibility
- Vulnerable populations affected

### Model Lifecycle Management

**Development Phase**
- Ethics review checkpoint
- Bias testing requirements
- Documentation standards
- Stakeholder consultation

**Deployment Phase**
- Gradual rollout (canary deployment)
- Performance monitoring
- User feedback channels
- Incident response plan

**Monitoring Phase**
- Continuous fairness metrics
- Drift detection
- Regular audits
- Retraining triggers

## Case Studies

### Case 1: Hiring Algorithm Bias

**Problem**: Amazon's resume screening tool penalized women's resumes

**Root Causes**:
- Training data from male-dominated tech industry
- "Women's" clubs and activities ranked lower
- Historical bias encoded in successful resumes

**Lessons**:
- Training data reflects historical bias
- Proxy discrimination through activities
- Need for fairness constraints
- Importance of intersectional testing

### Case 2: Healthcare Disparities

**Problem**: Algorithm under-recommended care to Black patients

**Analysis**:
- Used healthcare costs as proxy for health needs
- Black patients historically receive less care
- Lower costs didn't mean lower needs

**Fix**:
- Direct health measures instead of costs
- Race-stratified model evaluation
- Regular fairness audits

### Case 3: Facial Recognition Failures

**Problem**: Higher error rates for darker-skinned women

**Findings**:
- Training data predominantly light-skinned men
- Intersection of race and gender bias
- Real-world harm in law enforcement use

**Response**:
- Some cities banned police use
- Companies improved training data
- Increased focus on demographic testing
